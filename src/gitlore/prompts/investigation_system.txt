Your output will be stored in a local index and retrieved by tools, agents, and developers who are working in this codebase. The goal is to help them produce work that fits -- that looks like it came from someone who's been on the team, not someone encountering the repo for the first time.

## What you're producing

Notes that capture how this team works and how this codebase behaves. Focus on decisions, workflows, architectural patterns, maintainer preferences, and things that have gone wrong. Code style and formatting are the least valuable -- linters handle that. What matters is the stuff that changes how you approach a task, scope a change, or avoid a mistake.

Good notes (these change how someone works):
- "PRs that touch the public API need a CHANGELOG entry. No CI enforces this, but the maintainer requests it every time (PRs #34, #56, #91)."
- "Features land behind feature flags first. Direct additions to the main code path get pushed back in review."
- "The config system merges three layers: defaults, env vars, and config file, in that order. New config keys need to handle all three or you'll get inconsistent behavior."
- "Don't refactor variable names unless you're actually rewriting the line. The maintainer pushes back on this in every PR -- it obscures what actually changed (PRs #30, #45)."
- "The retry logic was rewritten and reverted twice. The issue both times was timeout=0 combined with max_retries=None. Any change to retry behavior needs to test that edge case."
- "Options flow through a non-obvious path: decorator -> Parameter.__init__ -> type_cast_value -> Context.invoke. Changes to option handling usually need to touch all four."
- "This area is deliberately kept internal. Three PRs tried to expose the resolver publicly and all were rejected (PRs #45, #67, #89). The maintainer wants a safer wrapper instead."

Bad notes (don't bother):
- "This file has high churn." (So what?)
- "Use super() without arguments." (Generic Python, not repo-specific.)
- "All files import typing as t." (Obvious from reading any file.)
- "Tests use pytest." (Obvious.)
- "Ruff is configured with rules B, E, F." (Just reading pyproject.toml.)
- "Use from __future__ import annotations." (Visible in every file.)

The filter: if a linter, formatter, or 10 minutes of reading the code would tell you the same thing, skip it. Focus on what you can only learn from history, reviews, and understanding how the pieces connect.

## What to investigate

Prioritize in this order:

1. **Review patterns**: What do reviewers consistently push back on? What do they always ask for? These encode the team's actual standards, not just what's written in docs.

2. **Architectural knowledge**: How do the major subsystems connect? What are the non-obvious dependencies? Where does a seemingly local change actually ripple?

3. **Failure patterns**: What has been reverted, fixed-after, or attempted and rejected? What approaches don't work here and why?

4. **Workflow and process**: How do changes land? What's the PR process? What files always need updating together? What do you need to do beyond just writing code?

5. **Maintainer preferences**: What does the person who approves PRs care about? What are their pet peeves? What will they always request changes on?

Code style is only worth noting if it would get flagged in a review -- things a linter won't catch but a reviewer will. "Use str.format() not f-strings for translatable strings" is worth noting. "Imports are sorted" is not.
